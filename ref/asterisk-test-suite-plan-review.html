Context

Earlier this session we manually diagnosed a real production misconfiguration: .env had ASTERISK_WSS_URL=wss://asterisk.velents.ai:8089/ws, but port 8089 is completely unreachable externally — kubectl get svc/ingress -n velents showed the real routing is the asterisk-wss Service (port 80 → targetPort 8088) fronted by an nginx Ingress on host asterisk.velents.ai (80/443), i.e. TLS terminates at the ingress and forwards plaintext WS to the pod's 8088. Nothing routes to 8089 externally.

The working URL is wss://asterisk.velents.ai/ws (default 443), confirmed via a 426 Upgrade Required response carrying Server: Asterisk/GIT-master-. The user fixed .env and asked for a repeatable test suite so this class of bug — config says one thing, actual K8s routing does another — gets caught automatically going forward.

Before designing anything new, I checked what already exists (scripts/audit-call-pipeline.sh, deploy/README.md, deploy/SIP_GO_LIVE_RUNBOOK.md), actually ran the existing script read-only against the live cluster, and read the ref/ directory (platform architecture + product-roadmap docs) to ground the plan in the real, current state of both the deployment and the broader product. That work surfaced three confirmed bugs in the existing test script and a full picture of what's built vs. planned across the platform — both captured below.

Correction — Inc 1's AI-audio piece

I originally wrote this as "missing, 3–5wk to build a pipeline." Wrong — two working AI-voice integrations already exist in sibling repos, just not bridged to Asterisk yet:

  • livekit-dispatcher (app.py) — FastAPI dispatcher with an ECall worker placing outbound calls via ElevenLabs' own SIP-trunk API — the "most-used" option.
  • livekit-outbound-caller (core/call_handler.py:52-56) — a LiveKit agent worker whose phone_dict already maps "innocalls" (the same SIP carrier Asterisk's PJSIP trunks use) to a LiveKit outbound-trunk ID. Caveat: the trunk-selection function is currently hardcoded to a different ID, with the innocalls branch commented out — present in code, not on the active path.
  • No code anywhere bridges Asterisk's AudioSocket tap (or its innocalls PJSIP trunk) to either — confirmed by a repo-wide search. Two parallel integrations against the same carrier, not one connected pipeline.

Net effect: the missing work is wiring, not a new AI-voice vendor integration — the table below reflects the revised estimate.

  • The teal group (Carrier → PJSIP → Dialplan → ARI/violet softphone) is the live, working call-control path — this is what today's audit-call-pipeline.sh gates (fixed per this plan) actually test.
  • The pink toggle node is the new addition: it's where a future "route via Asterisk vs. direct-to-carrier" decision would live, once one exists.
  • The amber group (livekit-dispatcher / livekit-outbound-caller) is today's live default AI-call path; the dashed line into the red-dashed AudioSocket node is the proposed alternative, and the innocalls mapping in livekit-outbound-caller is present in code but its selection branch is commented out — not a live end-to-end route yet.
  • The indigo group (velentsAgents' Integration module) already talks to both the Asterisk control-API and livekit-dispatcher — the natural place to eventually own both the AudioSocket bridge and the routing toggle, rather than inventing a fourth coordination point.

Decisions confirmed with the user

Wrapper lives as a project-scoped command (.claude/commands/audit-call-pipeline.md, not the global one); the narrow RTP range (10000-10003) is a known/accepted limitation → WARN, not FAIL; CONTROL_API_SECRET may be auto-read from .env as the BEARER default (explicit env var still overrides) — ARI credentials stay kubectl exec-only, never read from local .env.

All gate logic stays in scripts/audit-call-pipeline.sh — the single source of truth, runnable from a terminal, CI, or cron with zero Claude Code dependency. Add a thin wrapper command that only resolves inputs, invokes the script, and re-renders its output as a structured report — no gate logic duplicated in the command file.

1. Fix the three confirmed bugs

  • Gate A (audit-call-pipeline.sh:65-70): derive WSS_HOST once, preferring ASTERISK_WSS_URL from .env, falling back to ASTERISK_SIP_DOMAIN, then a hardcoded default. Replace grep -q '^asterisk-ws\.' with an exact match. Delete the independent re-derivation at line 79.
  • Gate A's RTP check (lines 55, 61-63): match individual NodePort UDP entries instead of a literal range string. Exactly the documented 4-port set → WARN, not FAIL; only FAIL if narrower or absent.
  • Gate D (line 124): fix the grep anchor from '^ Transport:' to '^Transport:'.

2. New gate — external reachability + .env config-drift guard

Inserted right after the (fixed) WSS TLS-cert gate — the gate that would have caught today's :8089 bug automatically: DNS resolution, HTTP/HTTPS reachability (lenient status-code PASS), the exact WSS-upgrade probe verified by hand this session (426 + Server: Asterisk header required), and a config-drift guard scanning .env for any *_WSS_URL/*_ARI_URL var that disagrees with the live ingress mapping. SKIP_EXTERNAL=1 opts out for offline/firewalled dev.

3. New gate — pod health

Restart-count / CrashLoopBackOff detection via kubectl get pods, plus a log scan that excludes confirmed-normal noise (trunk flapping, UNIX socket disconnects) before checking for genuinely fatal patterns.

4. New gate — ARI readiness + control-api /healthz cross-check

Closes the gap SIP_GO_LIVE_RUNBOOK.md item 3 flags as open: in-pod ARI readiness via kubectl exec, in-pod /healthz (labeled bind-only), and — the key new assertion — FAIL if /healthz is 200 but ARI readiness fails, naming the documented gap explicitly. Plus cross-cluster checks using BEARER against the real external ingress path.

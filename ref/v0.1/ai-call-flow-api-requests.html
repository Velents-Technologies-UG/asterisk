AI call flow — API request sequence

Inbound-call routing, the human-agent-assist copilot pipeline, and the AI-agent-handled call with human takeover — as three request/response flows, built from what's actually verified in the asterisk repo plus the platform-services and dossier docs in this directory. Updated this pass: the call-engine's location and the IVR FlowRunner's existence — both drawn as unresolved below in diagram 1 originally — are now confirmed against a real live inbound call test.

What this diagram assumes, not confirms

Four things below were drawn as forks or TBD nodes rather than resolved, because the answers materially change the architecture. Two of them — the call-engine's location and the FlowRunner's existence — are no longer guesses as of this pass.

fork

STT ingestion path for the human-assist pipeline — direct AudioSocket/externalMedia tap vs. bridging into a LiveKit room as a silent subscriber. Both are drawn in diagram 2; the earlier services-overview doc argued the direct tap is sufficient since this need is one-way. Changes whether this pipeline needs a room/SFU at all. Still open.

resolved this session

The call-engine (ARI/Stasis app) that every dialplan hook routes into is call-engine — a Node.js service, its own Kubernetes pod, deployed from the Velents-Technologies-UG/agent-hub repo. Proven live: a real inbound call was correctly picked up, and its DID resolved to the right tenant. One wrinkle remains — agent-hub's main branch is itself a Next.js frontend, with no trace of this backend at its root; see the companion Worker 2 doc.

resolved this session

The IVR FlowRunner that executes "route to AI vs. human vs. TTS based on this tenant's flow" exists — proven live: it loaded a real tenant Flow and walked its node graph from start to a transfer node. One open bug remains in the transfer step itself (ARI "Channel not found", ~600ms after the call arrives) — narrowed to a specific symptom, not fixed yet.

partially resolved

Tenant KB has two candidate sources that were never reconciled: velentsAgents' own Agents/KnowledgeBase model, vs. the standalone voice-agent service (a Weaviate KB API, despite the name — not an audio pipeline). Resolved for the reply_recommendation branch: reading text-agent's source confirms it calls voice-agent over HTTP (POST /kb/retrieve, one Weaviate collection per agent_id) — velentsAgents' own KB model is not used anywhere in that path. Still open for whatever would do tool_recommendation, which isn't built anywhere yet — diagram 2 now names voice-agent on the reply-rec branch specifically, and keeps Tenant KB (source TBD) on the tool-rec branch and in diagram 3.

not supported today

text-agent has no "generate N variations" capability — every chat/completion endpoint (/api/v1/sessions/{id}/process, DSPy /agent/chat) returns one response, and it discards whatever similarity score voice-agent might return, keeping only the retrieved text. reply_recommendation as originally proposed assumes two capabilities — N variations and a KB-match % — that would both need to be added upstream before this flow works as described.

confirmed, not new

MongoDB is already a live platform dependency, not a new technology — text-agent runs an async Motor client against it for session/conversation persistence (its own mlapi database). Using Mongo for AI-copilot task results would be a new database/use case on existing infrastructure, not a third stack to operate — still worth deciding whether it shares text-agent's instance or gets its own.

1 — routing

Inbound call → tenant IVR → queue decision

Where a call lands before any AI copilot logic runs: trunk → dialplan → call-engine → tenant IVR → one of TTS-only / human queue / AI queue. The call-engine and FlowRunner nodes are now proven-live (teal), not TBD/gap (amber/red) — a real inbound call test walked exactly this path up to the transfer step.

2 — human-agent assist

AI copilot request flow (human agent on the call)

Per-tenant feature toggles (sentimental_analyses, client_intention, tool_recommendation, reply_recommendation, ...) are sent as one config array to two places at call start — the ingestion/STT service and the velentsAgents backend. Disabled features (e.g. question_to_ai_recommendation: false) never fire a webhook. Raw call audio reaches the agent over the existing WebRTC/WSS softphone path — it never enters this pipeline. The reply-rec branch below is grounded in the real text-agent service, not assumed — see the "confirmed"/"not supported today" callouts above for what that changed.

3 — ai-agent handled

AI-agent call, human monitoring, and takeover

Same sentiment/intent analysis as diagram 2 runs alongside the AI's own STT → LLM → TTS loop, pushed to a human as a live transcript with tools-used and analysis. The human can optionally listen in — reusing the same ARI externalMedia snoop primitive already needed for supervisor whisper/barge — before deciding to take the call.

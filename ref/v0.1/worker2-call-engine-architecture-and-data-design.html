Worker 2 (call-engine) — architecture & data-ownership design

What call-engine actually does, why its deploying repo is architecturally a frontend, a tech-stack comparison for where its live event loop should run, and the resulting decision on data ownership as more clients share the call-center feature.

Repo Velents-Technologies-UG/agent-hub (deploys it) · e:\Projects\Velents\asterisk (consumes it) v0.1 — new doc
01
Why this doc exists

The feature-gap plan had an open question for weeks: where does the ARI/Stasis "call-engine" that every Asterisk dialplan hook routes into actually live? A real end-to-end call test this session finally answered it — it's call-engine, deployed from the Velents-Technologies-UG/agent-hub GitHub repo. That answer immediately raised a bigger one: agent-hub's own main branch is a Next.js frontend, with no trace of this backend service at its root. That mismatch — and the fact that more clients, not just velentsAgents, will eventually want to use the same call-center feature — is what this doc works through.

02
What call-engine does on an inbound call

In plain terms: it's the always-on process that Asterisk hands every live call to, and which decides what happens to that call. Asterisk's own dialplan does almost nothing itself — it recognizes an inbound call and hands the channel off:

NoOp(inbound ${CALLERID(num)} -> ${EXTEN})
Set(__CALLED_DID=${EXTEN})
Stasis(call-engine, inbound, ${EXTEN})
Hangup()

Confirmed directly from the service's own boot log this session:

ARI connected url=http://asterisk:8088 app=call-engine
ARI Stasis app 'call-engine' subscribed and ready

The sequence, step by step

  1. StasisStart arrives with the call's args. Confirmed: Flow StasisStart ... flow=forward-to-personal-test caller=voylo_locat did=+966115030505 source=inbound.
  2. DID resolution — an HTTP call to the central DID registry to find which tenant owns the number and what handles it. Confirmed: inbound DID '+966115030505' -> tenant=testCallCenter flow=forward-to-personal-test.
  3. Branch: Flow vs. Queue — a DID either runs a scripted Flow or routes straight to a live human-agent ACD queue.
  4. Flow execution — flow-runner.js loads the tenant's Flow (a JSON node graph) and walks it. Node types: start, play, dtmf_collect, queue_enter, hangup, set_var, if, webhook, transfer, voicemail, business_hours, bridge_ai.
  5. Transfer execution — transferer.js dials/bridges to an agent, extension, or e164 number.
  6. Call-state tracking — a lifecycle state machine intended to feed persistence; currently disabled in this deployment (no DB configured), so nothing is actually saved, only logged.
Where it broke, precisely

~600ms after StasisStart, before the transfer completed: flow run failed (inbound forward-to-personal-test): "Channel not found" → immediate teardown. An ARI command against the channel — almost certainly the flow's first real action — was rejected because the channel had already stopped existing on Asterisk's side. A timing/race bug internal to the flow-runner, not the dialplan or PJSIP layer.

Other subsystems wired at boot

  • Queue dispatcher wired (ACD enabled) — human-agent queue routing, the parallel path to Flows.
  • Outbound router wired — agent-initiated outbound calls.
  • AudioSocket stub listening on 0.0.0.0:8090 — backs the Flow's bridge_ai node; still just a 3-second-tone placeholder, not a real AI-audio bridge.
  • Redis pub/sub — likely for pushing live call state to a frontend; a gateway component is explicitly disabled in this deployment.
  • Its own internal Control API on 127.0.0.1:8092 — a different API from deploy/control_api.py's trunk-CRUD API (same port number, different pod, coincidence). This one exposes agent-facing mid-call actions like /control/calls/:uuid/transfer.

API surface, summarized

Calls out: DID resolution (GET .../Did/Resolve/<number>), Flow fetch (GET .../Flows/<id>), call-event writes (currently no-ops), Flow webhook nodes (arbitrary, per-flow), and ARI commands to Asterisk itself.

Exposes: /control/calls/:id/transfer (and likely hold/hangup) for agent-triggered mid-call actions.

03
The architecture mismatch

agent-hub's own package.json (root, main branch) is a plain Next.js frontend — next dev/build/start, dependencies like @dnd-kit and @fullcalendar. No ARI client, no WebSocket server, nothing resembling call-engine.

The one real clue

A commit in that repo's own history: "refactor(voice): remove Asterisk/Twilio telephony runtime from the UI." This backend likely used to live bundled inside the frontend and was split out at some point — but the deployed Jenkins build checks out a different branch (claude/innov2-sip-auth-error-pjgbng, not main), so whether that split finished, or landed anywhere permanent, is still unclear.

04
Tech-stack comparison — the live call event loop

"Worker 2" is really two kinds of work: provisioning/lookup APIs (normal request/response REST — a good fit for velents_integrations, a Laravel app) and the live call event loop — holding an open ARI connection, reacting in real time, sub-second timing sensitivity. That second half needs a long-running, event-driven process, which is why the current build is Node.js, not PHP. The comparison below is specifically for that second half.

StackPerformanceEffortEst. timeKey risks
Node.js — relocate current code to its own dedicated repo Proven live this session — ARI connect, Stasis subscribe, DID resolve, Flow load all worked against real production traffic. Low — no rewrite, extraction + CI/CD + one known bug fix 1–2 wk Inherits whatever else is unfinished/undocumented (disabled DB writes, disabled gateway) — needs auditing.
Node.js — clean rewrite, same language Same ceiling, cleaner internals. Medium-High — full rewrite of ARI client, flow-runner, transferer, resolver 4–8 wk Re-introduces bugs already solved once.
PHP + Laravel Octane (Swoole) inside velents_integrations Swoole gives a real async loop — capable, but ARI-for-PHP is a small, unproven ecosystem. High — new runtime mode, ARI client/engine built from scratch 6–10 wk Little Octane experience on the team; small community for support.
PHP + custom ReactPHP daemon (no Octane) Workable async I/O, hand-rolled reconnect/event handling. High — same from-scratch build, plus your own daemon supervision 6–9 wk Bespoke infra = its own class of subtle bugs (this session alone found four).
Python (ari-py + asyncio), new dedicated service Solid async support, smaller/less-maintained ARI ecosystem than Node's. High — full rewrite, new language for this domain 6–10 wk Smallest community of these options for ARI specifically.
Recommendation

Don't rewrite — relocate. The current Node.js code got 90% of the way there, live, this session, against production Asterisk. That's a small, fixable bug in working code, not evidence the stack is wrong. Rewriting elsewhere to satisfy "it should live somewhere else" throws away something that already works, for an ecosystem with less Asterisk/ARI tooling support.

05
Data-ownership design (decided)

velents_integrations will eventually serve more than just velentsAgents — other backends, for other clients (e.g. safha), sharing the same call-center feature. That forces a decision, per piece of data: tenant DB (owned by whichever backend serves that client), or a shared, backend-agnostic telephony store?

Precedent already in place

Two Postgres databases already exist, both named velentsagents, on two different RDS instances — a working example of roughly this split. PROD has sip_trunks/ps_endpoints/ps_auths/ps_aors/ps_identify — realtime PJSIP config Asterisk reads directly. NONPROD has did_registry/flows/queues/tenants — tenant/business data owned by velentsAgents' Laravel app.

The decision rule

Does Asterisk/call-engine need to read this at call-time, fast, without depending on a specific tenant's backend being up? Yes → shared telephony config, via velents_integrations. No, it's about the tenant's business → stays in that tenant's own backend.

Shared (velents_integrations)Tenant-owned (velentsAgents, safha's BE, ...)
DID registry — number → tenant + Flow/queue (must be globally unique across clients) Flow authoring — drafts, versions, edit history, unpublished changes
SIP trunk/endpoint/identify config (already the case today) Agent profiles, business rules beyond routing, CRM/customer context
Current published Flow (runtime definition only, not history) Conversation history, analytics, billing
Queue definitions for live routing (or a synced, low-latency copy) Anything gated by that backend's own auth/permissions model

The synced-copy model — decided, with refinements

For Flows (and anything else a tenant backend authors into the shared store): push a synced copy on publish, not a live fetch, not a webhook/polling scheme.

  • Trigger — explicit, on publish. A tenant backend calls a velents_integrations write endpoint right when something is published (e.g. POST /telephony/flows/{tenant}/{flow_id}). Publishing is already deliberate and low-frequency — no webhook infra or polling needed.
  • Read path — call-engine reads through velents_integrations' API, never the shared DB directly. That's the whole point: validation and constraints get enforced once, not re-implemented by every consumer.
  • Constraints enforced on every write — DID uniqueness globally, across every tenant/client; Flow schema validation (valid graph, no dangling references); a DID's referenced Flow must exist and be published before the mapping is accepted.
  • Failure handling — call-engine is never blocked by an unreachable tenant backend; it keeps serving the last synced copy until the next publish succeeds. One client's backend outage can never take down another client's live calls.
Why synced-copy over live-fetch

Live-fetch (today's model) is simpler and always current, but couples call-time reliability to every backend's uptime — exactly the cross-service dependency this design exists to avoid. Synced-copy costs an extra push at publish time and a brief window of possible staleness, in exchange for call-time reads never depending on any one tenant backend being reachable. Given the whole point is supporting multiple independent backends sharing one call-center feature, that tradeoff was the deciding factor.

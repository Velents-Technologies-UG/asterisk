Context

Earlier this session we manually diagnosed a real production misconfiguration: .env had ASTERISK_WSS_URL=wss://asterisk.velents.ai:8089/ws, but port 8089 is completely unreachable externally. The working URL is wss://asterisk.velents.ai/ws (default 443), confirmed via a 426 Upgrade Required response carrying Server: Asterisk/GIT-master-. The user fixed .env and asked for a repeatable test suite so this class of bug — config says one thing, actual K8s routing does another — gets caught automatically going forward.

After that plan was approved and shipped, the session kept going with a real end-to-end test: onboarding a brand-new inbound SIP trunk (a carrier called Voylo) all the way through a DB migration, a DID-registry row, and a Flow — then placing real, paid international test calls to prove the whole pipeline. That test found four more real bugs, including one self-inflicted production incident, finally resolved the call-engine's location, and led to a real architecture decision about where call-routing logic and its data should live as more clients start sharing this feature — captured in a companion document.

Real end-to-end inbound-call test

Rather than stop at "the config looks right," the session built a real test fixture — a new carrier trunk, a DID, and a Flow that transfers to a personal number — and placed real calls against it.

  1. Built the fixture via a checked-in DB migration: Voylo trunk (source-IP trust only), DID +966115030505 → tenant testCallCenter, a Flow that transfers to a personal number.
  2. First call failed instantly (CHANUNAVAIL in 14-64ms) — Asterisk logged "No matching endpoint found." This was Bug 7: IP-identification was silently broken platform-wide.
  3. Fixing Bug 7 required a process restart, which surfaced Bug 9 (stale network path from an earlier crash/rebuild — itself caused by Bug 8, a regression from this same session's /healthz fix). All three fixed/worked around before re-testing.
  4. Next attempt passed the network and IP-matching layers cleanly, but hit a 401 Unauthorized — Bug 10: a carrier that correctly never sends credentials, rejected by an auth requirement the trunk-creation API forces on every new trunk.
  5. With all four fixed, a real call went all the way through Asterisk correctly — confirmed directly in call-engine's own logs: inbound DID '+966115030505' -> tenant=testCallCenter flow=forward-to-personal-test.
  6. The one remaining failure is inside call-engine itself, not this repo — an ARI "Channel not found" error ~600ms after the call arrives, before the transfer completes. Narrowed to a specific, reproducible, cheap-to-retest symptom (about 1 second per attempt now, vs. 8-50 seconds before).
